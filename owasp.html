L'injection SQL, noSQL, OS et LDAP

Introduction :

L’attaque par injection correspond en fait à l’insertion d’un script, bout de code ou requête malveillante dans une application web (ou un programme). Le but d’une telle attaque peut être multiple, suivant les intentions du hacker : il peut chercher à afficher des valeurs stockées à l’intérieur des différentes tables d’une base de données, il peut vouloir connaître le nom des tables afin de pouvoir les utiliser à son avantage, et, dans le pire des cas, il peut supprimer certaines tables voire la base complète définitivement. Afin de comprendre ce concept, nous allons commencer par expliquer ce qu’est la plus connue d’entre elles, l’injection SQL. Cette faille très connue permet d’exploiter des failles de sécurité sur une application web (ou autre, mais on retrouvera plus facilement une base de données associée à une page web) qui dialogue avec une base de données associée. Le but de l’injection SQL est d’injecter dans une requête SQL un morceau de requête qui, par exemple, n’a pas été géré en tant qu’exception au niveau du back-end du site et qui pourra de ce fait dialoguer avec la base. Il peut également être possible de mettre à jour les données de la base afin de la corrompre. Tout cela se fait via tous les types d’input qui peuvent être présents sur le site (barre de recherche, authentification, accès au compte client…). Voyons dans un premier cas les trois types d’exploitations SQL qu’il existe.

Injection SQL

Il existe deux grandes familles d'injection SQL :
1. Injection SQL in-band :

Le type d’injection SQL en bande est un processus par lequel une personne malveillante va pouvoir en même temps pouvoir injecter du code dans un site web / une application, ainsi que d’afficher et donc avoir accès aux résultats. On distingue deux types d'attaques différents :

Attaque basée sur l’erreur :
Amener une base de données à renvoyer une erreur peut permettre d’afficher des noms de tables / de champs dans une table et, au final, à donner une topologie de la base de données à une personne malveillante. Cette attaque est surtout utilisée au moment d’afficher des éléments (liste sous forme de tableau ou de chaînes de caractères). Toujours en se plaçant dans le cas d’un site étant codé en PHP pour le lien avec la base, on peut par exemple se retrouver avec un affichage d'erreur indiquant le nom d'une des tables.

2. Injection SQL inférentielle (ou dite à l’aveugle) :

Ce type de requête SQL est utilisé lorsqu’une requête n’est pas directement transmis à une page Web. Elle permet d’envoyer des requêtes à une base de données afin de recueillir les informations qu’elle contient. On distingue encore une fois deux types d’injections SQL inférentielles :

Attaque booléenne :
Grâce à une attaque booléenne, un attaquant va juger de la protection d’une base face à une injection SQL. Il est question d’entrer dans des inputs de texte (imaginons par exemple un input de texte pour saisir un identifiant ou un mot de passe dans une interface de connexion quelconque) ces deux lignes :
“Alfred’ AND ‘1’=’2”
“Alfred’ AND ‘1’=’1”
Si, lors de la deuxième requête, on a un résultat différent du premier ou que l’application fonctionne, cela révèle l’existence d’une vulnérabilité à l’injection SQL (nous verrons un peu après en détails la significaiton du ‘1’=’1’.
Attaque basée sur le temps :
En utilisant une fonction temporelle prédéfinie sur le gestionnaire de base de données, il est possible pour un hacker de déterminer la présence (ou non) d'une vulnérabilité dans l'application web. L'attaquant peut par exemple utiliser une fonction temporelle prédéfinie dans MySQL, sleep, et s'il constate que la requête prend effectivement une certaine pause après exécution, cela signifie qu'il y a bien une vulnérabilité.

INJECTION SQL - Faille ‘ OR 1=1;-- et ‘admin’; --

Comme dit précédemment, il est notamment possible de pouvoir s’authentifier sur le compte d’un utilisateur simplement en connaissant son pseudo. Pour cela, plusieurs solutions s’offrent à vous : tenter des pseudos très communs au hasard ([prénom], [prénom.nom], [prénom_nom]) ou alors des utilisateurs admin par défaut (admin, root…). Pour trouver le mot de passe associé, dans l’input du mot de passe, il suffit d’écrire ‘ OR 1=1 -- ou bien ‘ OR 1=1 --, ‘ OR 1 --, il y a des possibilités infinies pour ce type de requête. En effet, il est question d’altérer la requête SQL vers la base en la modifiant. Il faut bien comprendre ici que, si le site est par exemple codé en PHP, on retrouvera dans le code une requête de type ‘’SELECT id FROM utilisateurs WHERE nom = $nom AND password = ‘$mdp’’’.

Exploitation de la faille
Ce qu’on cherche à faire dans cette faille, c’est réécrire la requête en modifiant sa structure. Ainsi, le « mot de passe » rentré par l’utilisateur malveillant va changer la fin du code, qui va alors devenir ‘’SELECT id FROM utilisateurs WHERE nom = $nom AND password = ‘ OR 1=1;--’’. La requête a donc changé, et vérifie ici deux choses : on sélectionne un id d’utilisateur dont le nom est ‘$nom’ et le mot de passe est ‘ ’ (mot de passe vide), ou alors on regarde si 1=1. Etant donné que 1 vaut bien 1, le résultat, sans avoir besoin de connaître le mot de passe puisque l’une des deux conditions est vérifiée (la condition OU est validée), va nous authentifier en tant que l’utilisateur rentré (le « -- » à la fin met simplement la suite de la requête en commentaire, afin d’esquiver toute vérification éventuelle). Il est aussi possible d’exploiter cette faille au niveau du username en entrant ’admin’; -- afin de prendre possession de l’utilisateur admin et pouvoir se connecter sur le site (marche avec tout utilisateur si le nom est connu).

Pour tout type de requête SQL, il ne faut pas laisser la possibilité aux utilisateurs de pouvoir entrer du code SQL librement en sécurisant les inputs de texte.
Multiplication de requêtes et déclarations préparées :
Multiplier les requêtes précises permet également de limiter l’accès aux données si une faille venait à être trouvée. Imaginons une requête SQL qui va sélectionner toutes les informations d’un utilisateur en comparant le nom entré à la liste d’utilisateurs contenue dans une table de la base de données, on obtient alors une requête du type : ‘’SELECT * FROM utilisateurs WHERE nom = $nom’’. Cette requête va retourner tous les champs de la table utilisateurs si l’injection malveillante fonctionne. Cependant, sélectionner simplement l’id d’un utilisateur pour après exécuter une autre requête sur cet ID pour renvoyer les informations de l’utilisateur permet de sécuriser l’accès aux différentes données relatives aux utilisateurs : ‘’SELECT user_id FROM utilisateurs WHERE nom = $nom’’.
Il est également possible d’utiliser des déclarations préparées pour éviter les injections SQL. Une instruction préparée est un modèle de requête SQL, dans lequel il faut spécifier des paramètres à un stade ultérieur pour l’exécuter. Voici un exemple d’une déclaration préparée en PHP et MySQLi :
$query = $mysql_connection -> prepare("select * from utilisateurs where username = ? and password = ?"); $query->execute(array($username, $password));
Il est de toute façon préférable de toujours utiliser PDO ou mysqli en PHP, qui mettent à disposition de l’utilisateur des fonctions de contrôle des données.
Echappement des caractères spéciaux :
En complément à toutes ces préventions, il faut savoir que, de manière générale, il est difficile de déterminer si une chaîne utilisateur est malveillante ou non. Par conséquent, la meilleure façon de procéder est d’échapper les caractères spéciaux dans la saisie de l’utilisateur, c’est-à-dire d’étudier une chaîne de caractère en amont de l’exécution de la requête SQL afin de pouvoir la traiter et gérer les caractères non voulus. Ce processus, également utilisé pour la faille XSS (détaillée plus tard), permet d’échapper une chaîne de caractères avant de construire la requête dans PHP en utilisant la fonction mysql_escape_string(argument). Il est également possible d’échapper une chaîne de caractères dans MySQL en utilisant la fonction mysqli_real_escape_string(argument).

INJECTION noSQL

Les bases de données noSQL permettent de manipuler une quantité de données bien plus importante que ce que peut gérer une base SQL classique tout en ne souffrant pas de la latence dont une telle base souffrirait. Généralement, on parle d’association clé-valeur : une base de données noSQL qui regroupe, par exemple, des logs de connexion, ou des événements serveur, peut être un tableau de millions d’entrées, auquel on associe une clé pour avoir accès.

Stockage - Type de bases / Schéma d'attaque

Le stockage peut également être orienté document, c’est-à-dire sous la forme de documents de type XML ou JSON pour la plupart, mais également orienté colonne ou encore orienté graphe. Ces bases sont donc conçues pour pouvoir naviguer dans des quantités énormes de données sans ralentissement et ont d’ailleurs été adoptées depuis une dizaine d’années par la plupart des plus gros sites brassant des quantités énormes de données (Facebook, Amazon, Google entre autres). On compte dans les serveurs de base de données les plus connus : CouchDB, MongoDB ou encore Oracle NoSQL (utilisation de fichiers JSON pour stocker les données).

Ainsi, vous l’aurez compris, une base noSQL ne possède pas de réelle structure, à l’opposé d’une base de données SQL qui s’appuie sur des solides relations entre ses tables déterminées au départ par un diagramme UML.

Authentication Bypass :
Afin de contourner l’authentification, un hackeur va avoir recours à l’utilisation d’opérateurs d’égalité : $eq (equals), $ne (not equal) ou encore $gt (greater than), qui sont indiqués dans la documentation de MongoDB, et plus particulièrement ici à l’utilisation de $ne.
Pour faire simple, si on vient spécifier que le nom d’utilisateur et le mot de passe ne sont pas égaux à 1 (username[$ne] = 1 & password[$ne] = 1), cette requête est interprétée par la base MongoDB comme une demande de trouver tous les utilisateurs et les mots de passe qui ne sont pas égaux à 1.
Ainsi, dans la même lignée que la faille OR 1=1 en SQL, cela revient à spécifier un élément qui est toujours vrai et, si aucune sécurité n’est mise en place, une personne malveillante aura alors accès à la base en ayant contourné l’authentification, car la condition « pas égal à 1 » sera validée.
Pour plus d’informations sur le login Bypass, OWASP a publié un PDF regroupant la plupart des attaques connues (sous forme de POST, en utilisant les $ne…) : https://owasp.org/www-pdf-archive/GOD16-NOSQL.pdf
Javascript Injection :
Une autre façon d’effectuer une injection noSQL est d’utiliser l’opérateur $where, qui va permettre à un hackeur d’injecter du code contenant un morceau de JS dans la base. Bien que cet opérateur ne permette pas d’utiliser toutes les possibilités offertes par JS de façon arbitraire, il est quand même possible pour quelqu’un d’utiliser des fonctions intéressantes.
Par exemple, en utilisant la fonction find() sur la collection dont on désire retirer des informations dans la base de données, on aurait une ligne de cette forme : db.collection.find({$where : function() {return (this.name == $user_data)}}); . Cette ligne signifie qu’on veut afficher la liste des informations d’un utilisateur contenue dans la variable $user_data en allant la chercher dans la collection de la base sélectionnée.
En complément à cette injection Javascript, il est possible d’utiliser une fonction temporelle comme en injection SQL classique afin de mettre en pause le serveur de base de données pour savoir si la faille existe bien, à l’aide de la fonction sleep(n) (pour n = 10000, le serveur sera en pause pendant 10 secondes). Si on l’injecte dans la requête précédente, on obtient : db.collection.find( {$where : function() {return (this.name == ‘hi’ ; sleep(10000) ) } } ); . Vous aurez donc compris qu’on a quelques possibilités permises par cette fonction.

 Interfaces REST clients / serveur de base de données – Echappement et vérification :
Avant de nous intéresser au côté serveur et au backend de celui-ci, il faut déjà faire attention à l’interface client. En effet, les serveurs de bases de données noSQL type MongoDB ou CouchDB sont généralement accompagnées de clients RESTfull, qui permettent de faire le lien entre un (ou plusieurs) client(s) et le service Web. Une telle API REST permet donc à plusieurs clients pouvant avoir des systèmes d’exploitation / des architectures différent(e)s, de communiquer entre eux.
Il faut donc mettre en place un moyen de sécuriser les INPUTS : par exemple, MongoDB possède des fonctions et autres features permettant de sécuriser la création de requêtes. Une autre chose à faire (pour tout type de problème d’injection) est de bien utiliser la règle du « least privilege », qui signifie qu’un utilisateur ne doit posséder que les droits et privilèges qui lui conviennent et uniquement ceux-ci. Il faut donc faire des séries de vérification, mais également d’échappement de caractères suivant le type de base en place, le type de langage utilisé pour dialoguer avec (PHP, Javascript, Python, NodeJS, Pearl…).
En dernier lieu, il faut aussi utiliser des requêtes préparées pour les inputs, afin que l’utilisateur ne puisse pas injecter le contenu qu’il veut depuis l’interface client : utilisation d’expressions régulières, hachage des valeurs entrées / des variables.

INJECTION OS

L’injection en ligne de commande (ou injection OS / OS Command Injection en anglais) est un type d’injection grâce à laquelle une personne malveillante va pouvoir, à l’aide de caractères spéciaux, modifier une commande initialement demandée.

Pour être plus précis, cela permet, lorsqu’une application ou ligne de commande construit une commande système utilisant des entrées (input) qui peuvent être influencées par l’utilisateur, de transformer la requête à l’avantage d’un hackeur. Cela vient souvent du fait que le software en question ne tient pas compte des caractères spéciaux et n’annule pas leur effet, entraînant ainsi des failles de sécurité.

Ce problème d’injection OS peut être d’autant plus retors si la personne malveillante parvient à avoir accès à des commandes systèmes privilégiées (droits administrateur) ou s’il a accès à un processus utilisant des commandes système dans lequel il peut y injecter n’importe quel code.

On compte deux types d’injections de commandes OS :
Inputs extérieurs non sanitarisés :
L’application s’apprête à exécuter un programme complet au service de celle-ci, et cette application accepte des inputs extérieurs comme arguments à ce programme (ajout d’arguments manuellement par l’utilisateur donc). Par exemple, si un programme utilise l’appel system(« nslookup [HOSTNAME] »), l’utilisateur doit donc entrer un hostname. Si l’échappement des séparateurs n’a pas été mis en place, le hackeur peut alors mettre des séparateurs avant la commande nslookup et exécuter le code voulu, de la même façon que pour la faille d’injection SQL étudiée précédemment.
Application utilisant une commande de source externe :
L’application a recours à un input qui sélectionne quel programme lancer et quelle commande utiliser, et elle redirige ensuite la commande vers le système pour exécution. Ainsi, si par exemple une application a recours à la ligne exec([COMMAND]), où ladite commande vient d’une source externe (et donc non vérifiée par des échappements de caractères entre autres), alors le hackeur qui contrôle cet argument peut exécuter de façon arbitraire n’importe quelle commande et la transmettre au système complet.

INJECTION OS – Protection contre les Failles

 Echappement / Remplacement de caractères :
On ne le dira jamais assez, la première chose à faire pour éviter les problèmes d’injection est d’échapper les caractères spéciaux, afin qu’une personne malveillante ait une erreur en voulant utiliser un séparateur, une mise en commentaire… tout ce qui concerne l’injection d’arguments qui ne devraient pas pouvoir transiter vers les données sensibles. Il faut donc « sanitizer », c’est-à-dire nettoyer les inputs via les commandes suivantes :
En PHP : escapeshellarg(« paramètre ») | escapeshellcmd() | preg_replace(« /[^a-z0-9\-.]/I”,””, paramètre)
Pour l’utilisation de l’échappement dans d’autres langages, voir le lien suivant : https://www.immuniweb.com/vulnerability/os-command-injection.html
Certaines invites de commande comme PowerShell sous Windows échappent nativement les caractères spéciaux, empêchant l’utilisation de tels caractères autrement que dans une chaîne de caractères (ils deviennent alors inutilisables et sont simplement considérés comme du texte) :
Web Application Firewall (WAF) :
Utiliser un firewall d’application web peut s’avérer efficace pour éviter les vulnérabilités pendant un temps, mais cela ne constitue pas une solution à long terme : si l’application en question peut avoir un patch de correctif de sécurité, il faut privilégier son installation plutôt que l’utilisation du firewall.
Laisser les programmes en local :
Si un programme doit en appeler un autre ou qu’il doit utiliser des privilèges / commandes système, l’utilisateur ne doit en aucun cas y avoir accès pour éviter l’injection. Si toutefois il est absolument nécessaire qu’il y ait accès, pour entrer un paramètre manuellement, il faut alors mettre en place un échappement de caractères comme décrit au-dessus.

INJECTION LDAP

Introduction au LDAP :

Pour comprendre cette partie, nous allons dans un premier temps définir ce qu’est le LDAP. Le protocole Lightweight Directory Access Protocol (LDAP) est un protocole logiciel constituant une norme pour les systèmes d’annuaires, c’est-à-dire tout ce qui concerne les bases d’informations sur les différents utilisateurs d’un même réseau, grâce au protocole TCP/IP.

Il rend donc capable n’importe quel appareil sur un réseau de trouver d’autres ressources localisées elles-aussi sur ce réseau : on retrouve notamment d’autres ordinateurs ou des fichiers envoyés à travers celui-ci. Ce protocole permet également de définir la méthode d’accès aux données d’un serveur du côté client. Le serveur fournira ainsi à l’utilisateur des méthodes qui lui permettent par exemple de se connecter, de comparer des informations avec d’autres entités…

Introduction à l'injection LDAP :

L’injection LDAP concerne l’exploitation d’une application web qui construit des protocoles LDAP en fonction des informations entrées par l’utilisateur. Etant donné que le protocole LDAP utilise des requêtes qui requièrent l’utilisation de caractères spéciaux. Ils contrôlent donc la signification des requêtes envoyées au serveur et sont utilisables par un utilisateur qui peut, s’il est malintentionné, changer la valeur de la requête initiale pour y insérer un bout de code malicieux et ainsi faire une injection sur le serveur. Un hackeur peut ainsi extraire des données potentiellement sensibles depuis le directory LDAP d’un serveur d’une entreprise ou autre.

Pour mieux comprendre comment fonctionne l’injection LDAP, commençons par nous intéresser à la syntaxe utilisée. Il faut savoir que la recherche LDAP fonctionne via une combinaison de filtres (composé d’une paire clé-valeur à la façon d’une base noSQL) et d’opérateurs de comparaison dont voici des exemples :

(cn=Jean*) : cette ligne matche les entrées où le « common name » (prénom) commence par « Jean »
( ! ( cn=*DUPONT) ) : matche les entrées où le nom de famille ne vaut PAS « DUPONT »
( & (cn = J*)(cn=*DUPONT) ) : matche les entrées où le common name commence par un « J » ET finit par « DUPONT »

Ainsi, comme dit précédemment, les caractères spéciaux utilisés ont beaucoup de sens et il est facile de voir que cela peut vite devenir un problème si un utilisateur s’en sert à des fins malicieuses. De plus, comme pour toutes les autres injections, on peut tout de suite voir qu’il est possible de faire de l’injection à l’aveugle afin de voir si la réponse renvoyée est vraie, auquel cas il existe bel et bien une faille, et ensuite il est possible de récupérer un nom d’attribut ou de variable en cherchant un peu pour ensuite attaquer la base.

Les attaques fonctionnant comme pour toutes les autres injections, nous ne verrons ici que l’authentication Bypass (attaque la plus classique). Les autres attaques de type Information Disclosure, Blind Injection (Attribute discovery, Booleanization, Character Set Reduction) sont disponibles sur le site : https://www.netsparker.com/blog/web-security/command-injection-vulnerability/

INJECTION LDAP – Attaques

 Authentication Bypass :
Comme vu précédemment, l’utilisation de filtres en association avec les opérateurs *, & et ! notamment permet de faire des requêtes complètes sur le serveur LDAP. Dans le cas de l’authentification à un serveur LDAP, on aurait une requête de ce type :
filtreAuthentification = ‘’ ( & (userID= ‘’ + enteredUser + ‘’)(password = ‘’ + enteredPassword + ‘’ ) ) ‘’
Une fois cette requête entrée, si elle est vraie, cela signifie donc qu’il existe une telle personne dans le directory et la personne sera authentifiée. Ainsi, si un hackeur souhaite s’authentifier sur le serveur, il doit simplement trouver une condition qui est toujours vraie comme pour les autres injections, ce qui donnerait la requête suivante :
filtreMalicieux = ( & (userID=*)(userID=*) )( | (userID=*)(password=blablabla) )
La requête ci-dessus signifie : si l’ID utilisateur vaut n’importe quel caractère et qu’il vaut n’importe quel caractère OU BIEN que l’ID utilisateur vaut n’importe quel caractère et que le mot de passe vaut blablabla alors l’utilisateur pourra être authentifié. La première partie de la requête étant tout le temps vraie, la personne malveillante sera donc connectée sans connaître ni le nom d’utilisateur ni le mot de passe requis.

INJECTION LDAP – Protection contre les failles :

 Echappement des caractères spéciaux :
Deux méthodes sont principalement utilisées pour se prémunir de la faille par injection LDAP, comme pour les autres types d’injection : procéder à l’échappement des caractères spéciaux, afin que seules les requêtes déjà préparées puissent les utiliser vers le serveur. Dans le cas de l’injection LDAP, il convient d’échapper au-moins : ! & ( ) et, surtout, *.
Framework / librairies sécurisés :
La deuxième méthode n’a pas été énoncée avant mais concerne toutes les injections : il s’agit de l’utilisation de framework ou de librairies déjà implémentées au niveau sécurité afin de sécuriser les requêtes sans avoir à construire plusieurs fonctions au backend en ne partant de rien. Cette solution est utile car ces bibliothèques contiennent déjà tout un set de fonctions utiles déjà implémentées qu’il faut simplement réutiliser avec les bonnes variables pour sécuriser le serveur.
Broken authentication

Introduction :

Broken Authentication est un terme qui désigne les failles concernant la gestion des sessions et la gestion des identifiants. Les attaquants emploient une grande variété de stratégies pour obtenir des informations confidentielles dans le but d’usurper des identités, revendre des bases de données ou récupérer des numéros de carte bancaire. Ils peuvent cibler tout un site pour récupérer la base de données des utilisateurs du site en question ou bien cibler une personne en particulier pour récupérer des informations critiques le concernant.

Broken authentication - Exploitation des failles

Credential Stuffing :
Une attaque Credential Stuffing est une technique brute force qui est basée sur l'utilisation d'identifiants compromis, d'identifiants contenus dans des bases de données piratées pour accéder à des comptes utilisateurs.
En effet, en utilisant cette attaque, les hackers partent du principe que les utilisateurs utilisent régulièrement le même mot de passe pour différents comptes. Pour renforcer le brute-forcing, ils utilisent généralement des botnets (un ensemble de machines interconnectées qui sont composées de bot).
La seule manière de se protéger de ceci est de ne pas réutiliser le même mot de passe, ou des mots de passe faibles très connus.
Ci-dessus un schéma montrant le fonctionnement de l'attaque par credential stuffing.
Le seul moyen de se protéger contre des attaques brute-force est d’implémenter une fonction qui limite le nombre d'essais par machines.
Password Spraying :
Le password spraying est aussi une attaque basée sur du brute-force, mais, contrairement à l’attaque précédente où un hacker essayait différents noms de comptes et différents mots de passes, ici le hacker brute-force différents comptes avec un mot de passe compromis ou connu.
Le but de cette attaque, à contrario avec la précédente, est d'être plus discret. En effet, de cette manière, l’attaquant se fera moins bloquer en attaquant une fois plusieurs comptes.

Broken authentication - Protection contre les failles

Control Session Length :
Toute application web qui gère des sessions devrait gérer le "temps de vie" des sessions de ses utilisateurs, de manière à ce que ne soit pas embêtant pour l’utilisateur tout en ayant un temps de vie relativement court pour limiter les risques de détournement de sessions.
Rotate and Invalidate Sessions IDs :
Toujours dans le but d’éviter l’usurpation de session, il est impératif de fermer une session une fois qu’elle est finie et donc supprimer les id liés à cette session pour faire en sorte qu’elle ne soit plus valide.
Don’t put Session IDs in URLs :
Une autre mauvaise pratique est de faire passer les id de sessions dans les url, qui pourraient ressembler à ceci : http://badwebsite/jsessionid=1234
Si une application web utilise cette pratique, il est nécessaire de la modifier car un attaquant pourrait en profiter pour effectuer des attaques telles que Session ID Fixation ou Session ID URL Rewriting
Implement Multi Factor Authentication (MFA) :
Multi Factor Authentication ou plus souvent 2FA (Two Factor Authentication) est un dispositif qui demande deux système d’authentification différents lors de la demande d’accès au compte, par exemple en entrant identifiant et mot de passe, le site demande alors un deuxième token qui certifie que vous êtes la personnes autorisée à accéder au compte. Le deuxième facteur d’authentification peut être de toute sorte : on retrouve notamment les YubiKey, le numéro de téléphone, une passphrase supplémentaire, etc…
Au delà de renforcer l’accès au compte, cela permet aussi de protéger les données dans le cas de la compromission de la base de données de l’application web.
Don’t Permit Weak Passwords :
Il est aussi nécessaire de réguler le formulaire de création de mots de passes. Un mot de passe est considéré sécurisé s’il contient au minimum 8 caractères, un chiffre, une majuscule et un caractère spécial.
Don’t Store passwords in plaintext :
Un mot de passe ne doit jamais être enregistré dans une base de donnée en clair, il est toujours nécessaire de hasher ce mot passe avec un algorithme de hash sécurisé comme sha-512 par exemple.
Si les mots de passe étaient stockés en clair, dans le cas d’une brèche, les conséquences seraient terribles.
Use Breached Password Protection :
Il est possible et fréquent que des utilisateurs utilisent des mots de passes compromis, c'est-à-dire qui sont apparus dans une base de données récupérée par un hacker. Il est donc de bonne pratique de vérifier lors de la création d’un compte si le mot de passe fait partie des mots de passe compromis. Pour vérifier, il existe différents outils ou package qui offrent cette possibilité. Specops possède notamment une API qui permet cette fonctionnalité. Il est également possible de regarder sur des sites comme "Have I been Pawned" : https://haveibeenpwned.com pour s'assurer de la fiabilité de son adresse mail.
Implement Brute-Force Protection :
Il peut enfin être utile d’implémenter une protection contre le brute-force, en limitant le nombre d’essais par exemple.

XML External Entities (XXE)

Introduction :

Une attaque XEE pour XML External Entities, parfois appelée attaque par injection XEE, est une vulnérabilité touchant les programmes interprétant le langage XML. Elle se caractérise par la possibilité d’extraire des fichiers du serveur web, et extraire les informations de ce dernier. Elle peut ainsi mettre en danger celui-ci, en accédant, par exemple, à un fichier de configuration contenant des mots de passe, en copiant les fichiers de la base de données ou en récupérant le code source d’une application. D’autres conséquences fortes peuvent découler d’une attaque par XEE, comme la possibilité de réaliser des requêtes externes, des dénis de service, des scans de ports …

XML et sa structuration :
Dans le but de comprendre correctement XEE, nous allons voir quelques basiques de XML. XML est un langage très populaire dans l’échange de données. On le retrouve notamment dans les web services comme les API SOAP ou REST au même titre que du Json.
Pour interpréter les données d’un fichier XML, il est nécessaire pour une application de parser ces données pour obtenir son contenu et l’afficher selon sa structure. La plupart des parseurs XML sont vulnérables par défaut aux attaques par injection XEE, ce qui rend la vulnérabilité plus forte pour des développeurs non avertis.
Tous les éléments XML sont contenus dans des balises.
<foo>
Hello

</foo>
Attardons-nous un instant sur la structure des fichiers XML. XML est un langage de balises ce qui sous-entend que le contenu est séparé des instructions de structure du fichier. Il est ainsi possible de définir proprement des blocs de construction et des entités dans un fichier séparé : DTD (pour Document Type Definition).
L’attaque par injection XEE opère sur le DTD. C’est pourquoi nous allons nous attarder sur cette notation. Un DTD est lié au document XML en utilisant une déclaration DOCTYPE et pouvant être utilisé de deux manières différentes : directement dans le fichier XML ou bien avec une référence externe faisant appel au fichier DTD.
Un exemple de DTD :
<?xml version="1.0"?>
<!DOCTYPE librairie [
<!ELEMENT encyclopedie (personne*)>
<!ELEMENT personne (nom,prenom,publication+)>
<!ATTLIST personne sexe (H | F) "H">
<!ELEMENT nom (#PCDATA)>
<!ELEMENT prenom (#PCDATA)>
<!ELEMENT publication (#PCDATA)> ]>

Ce DTD définit alors la version du document et la structuration des éléments qui vont le composer :
- Pour chacun des éléments, on définit sa composition par <!ELEMENT nom_element (structure)>
- Pour chacun des attributs, on définit sa composition par <!ATTLIST nom_element nom_attribut (structure)>
- L'attribut sexe de la balise <personne> est, ici, défini comme étant un choix imposé entre "H" et "F". Les valeurs possibles sont séparées par des caractères '|'. Cet attribut n'étant pas indiqué comme obligatoire, s'il n'est pas précisé alors il prendra par défaut la valeur "H" (la valeur précisée entre guillemets après la liste)
- personne* signifie que dans un élément <encyclopedie> on peut trouver de 0 à plusieurs éléments de type <personne> à l'exclusion de tout autre élément.
- nom, prenom, publication+ signifie que dans un élément personne on doit trouver un et un seul (*) élément <nom>, un et un seul élément <prenom> et un ou plusieurs (+) éléments <publication> et dans cet ordre (comme imposé par la virgule).
- <nom>, <prenom> et <publication> sont des éléments qui peuvent contenir du texte comme l'indique #PCDATA.
Ainsi, nous pourrons construire le fichier XML de telle sorte que les données respectent cette structure, en faisant référence au DTD défini en « externe ».
<?xml version="1.0"?>
<!DOCTYPE encyclopdie SYSTEM "encyclopedie.dtd" [
<encyclopedie>
<personne datenaissance="1942-01-08" sexe="H">
<nom>HAWKING</nom>
<prenom>Stephen</prenom>
<publication>Une brève histoire du temps</publication>
</personne>
<personne datenaissance="1932-07-13" sexe="H">
<nom>REEVES</nom>
<prenom>Hubert</prenom>
<publication>L'Univers epliqué à mes enfants</publication>
<publication>Patience dans l'azur, L'évolution cosmique</publication>
<publication>Poussière d'étoiles</publication>
</personne>
</encyclopdie>


XXE - Exploitation des failles

Billion laugh attack :
Billion laugh attack est une attaque par déni de service (DoS). Il est possible alors pour l’attaquant de démultiplier un élément afin de créer un flux de données très grand et rendre l’utilisation de l’application impossible. Regardons un exemple de document XML qui peut être envoyé à la victime :
<?xml version="1.0"?>
<!DOCTYPE lolz [
<!ENTITY lol "lol")>
<!ENTITY lol2 "&lol; &lol; &lol; &lol; &lol; &lol; &lol; &lol; &lol; &lol;">
<!ENTITY lol3 "&lol2; &lol2; &lol2; &lol2; &lol2; &lol2; &lol2; &lol2; &lol2; &lol2;">
<!ENTITY lol4 "&lol3; &lol3; &lol3; &lol3; &lol3; &lol3; &lol3; &lol3; &lol3; &lol3;">
<!ENTITY lol5 "&lol4; &lol4; &lol4; &lol4; &lol4; &lol4; &lol4; &lol4; &lol4; &lol4;">
<!ENTITY lol6 "&lol5; &lol5; &lol5; &lol5; &lol5; &lol5; &lol5; &lol5; &lol5; &lol5;">
<!ENTITY lol7 "&lol6; &lol6; &lol6; &lol6; &lol6; &lol6; &lol6; &lol6; &lol6; &lol6;">
<!ENTITY lol8 "&lol7; &lol7; &lol7; &lol7; &lol7; &lol7; &lol7; &lol7; &lol7; &lol7;">
<!ENTITY lol9 "&lol8; &lol8; &lol8; &lol8; &lol8; &lol8; &lol8; &lol8; &lol8; &lol8;">
<!ENTITY lol10 "&lol9; &lol9; &lol9; &lol9; &lol9; &lol9; &lol9; &lol9; &lol9; &lol9;">
]>
<lolz>&lol10;</lolz>
On remarque ici la notation &lol10 : c’est en réalité la valeur de l’entité définie dans le DTD.
Le parseur va alors au départ afficher les 10 valeurs de l’entité lol10, soit 10 fois lol9. Puis pour chaque valeur lol9, le parseur affichera 10 valeurs lol8. Ce pattern va ainsi se répéter jusqu'à atteindre la première entité. C’est alors exponentiel, d’où l’appellation « Billions laugh attack », car des milliards d’instances du string « lol » sont alors créées, surchargeant la mémoire du parseur et créant alors un Dos du système.
External Entity Attack :
Comme nous l’avons vu précédemment, il est possible de créer un fichier DTD “externe” au document XML. Cela offre la possibilité à un attaquant d’envoyer des requêtes qu’il aurait créé en amont en modifiant l’URI du DTD, et ainsi permettant de récupérer des ressources locales, comme un fichier de mot de passe dans l’exemple ci-dessous.
<?xml version="1.0"?>
<!DOCTYPE encyclopedie [
<!ELEMENT encyclopedie ANY>
<!ENTITY xxe SYSTEM "file:///etc/passwd">]>
]>
<encyclopedie>&xxe;</encyclopedie>
Le document alors généré par ce code XML et retourné à l’attaquant incorporera les mots de passes du fichier /etc/passwd de l’application (grâce à l’utilisation de &xxe).
Bien évidemment, si l’attaquant connaît la structure des chemins de l’application, il peut récupérer d’autres fichiers, écrire des commandes d’exécution, scanner des ports, ou bien même créer de requêtes HTTP sur le réseau local selon le parseur utilisé par la victime.
<?xml version="1.0"?>
<!DOCTYPE encyclopedie [
<!ELEMENT encyclopedie ANY >
<!ENTITY xxe SYSTEM "file:///c:/boot.ini">]>
]>
<encyclopedie>&xxe;</encyclopedie>

<?xml version="1.0"?>
<!DOCTYPE encyclopedie [
<!ELEMENT encyclopedie ANY >
<!ENTITY xxe SYSTEM "http://192.168.0.1/secret.txt">]>
]>
<encyclopedie>&xxe;</encyclopedie>

<?xml version="1.0"?>
<!DOCTYPE encyclopedie [
<!ELEMENT encyclopedie ANY >
<!ENTITY xxe SYSTEM "http://localhost:99">]>
]>
<encyclopedie>&xxe;</encyclopedie>
Pour plus d’exemples d'injection XXE, il existe un github répertoriant des exemples de XML malicieux à injecter :
https://github.com/swisskyrepo/PayloadsAllTheThings/tree/master/XXE%20Injection

XXE - Protection contre les failles

Quand sommes-nous vulnérables ?
- Les interpréteurs XML dans l’application acceptent les DTD.
- Les interpréteurs XML acceptent les entités externes.
- L’application accepte des modifications XML, notamment de sources externes non identifiées.
Deactivate DTD :
Le meilleur moyen pour se défendre contre tous les types d’injections XEE est de simplement désactiver les DTD. Si cela n’est pas possible, il suffit d’update la librairie de son interpréteur XML, car les nouvelles versions de ces librairies désactivent automatiquement les DTD et les entités externes.
Par exemple, si on utilise XmlDocument, voici comment désactiver les DTD :
XmlDocument doc = new XmlDocument();
doc.XmlResolver = null;
Une liste des différentes fonctionnalités à ajouter au code selon tous les parseurs existants est disponible ici :
https://cheatsheetseries.owasp.org/cheatsheets/XML_External_Entity_Prevention_Cheat_Sheet.html
Set max characters :
Pour se défendre contre les DoS, notamment « Billion lauth attack », il est possible d’ajouter une valeur maximum des entités externes. Ci-dessous un exemple avec XmlReader :
settings.MaxCharactersFromEntities = 1024;
On peut également ajouter dans le code une valeur Timeout de la requête entrante et sortante, afin d'éviter un temps d’attente trop long de la réponse et éviter un DoS.
Block URIs :
Il est possible de bloquer les URI localhost telles que http://localhost, http://127.0.0.1, et file:// URIs. En utilisant notamment des fonctions telles que IsLoopback, on peut contrôler les flux DTD.
Dans un cas général, il est possible d’ajouter des filtres whitelist dans son code et une validation des entrées côté serveur afin de ne laisser passer que les documents XML souhaités.
Bonnes pratiques :
- Patcher/Upgrade les librairies et parseurs XML le plus souvent possible.
- Limiter la mémoire et les ressources du parseur XML.
- Vérifier si les entrées XML sont contrôlées avant parsing.
- Vérifier si les entités externes sont autorisées.
- Vérifier si les DTD sont autorisés par le parseur et si oui, ceux qui le sont ne doivent jamais être construits par des utilisateurs non définis.
Broken Access Control

Introduction :

Il est tout d’abord nécessaire de comprendre la différence entre les failles Broken Access Control et les failles Broken Authentication vues précédemment. Prenons un exemple afin de comprendre ce qui les différencie :

Imaginons une simple application web non sécurisée qui possède un tableau de bord administrateur exposé sans mécanisme d’authentification. Ici, c’est un problème de Broken Authentication, car le besoin d’authentification pour accéder à des données sensibles n’est pas implémenté.

Par contre, si cette fois ci il y a bien un mécanisme d’authentification, mais qu’il est toujours possible d’accéder à ce tableau d’abord en étant authentifié en tant qu’utilisateur classique alors c’est un problème de Broken Access Control.

Concrètement, ce sont des failles où les contrôles d’accès ne sont pas bien implémentés et permettent ainsi à des utilisateurs d’accéder à des ressources auxquelles ils ne devraient pas avoir accès.

Voici quelques vulnérabilités liées aux contrôles d'accès qu’il est commun de retrouver.

Broken Access Control - Failles

IDOR - Insecure Direct Object Reference :
IDOR ou Insecure Direct Object Reference est un faille qui intervient lorsqu’un site web utilise des inputs fournis par l’utilisateur pour accéder à des ressources. Cette faille est souvent associée à des élévations de privilèges horizontal, mais elle peut être également liée à des élévations de privilèges vertical.
Une élévation de privilège horizontal désigne le fait qu’un utilisateur quelconque accède au contenu d’un autre utilisateur quelconque.
Une élévation de privilège vertical quant à elle désigne le fait qu’un utilisateur avec des privilèges moindres accède à du contenu, des fonctions ou des applications d’un utilisateur qui possède plus de privilèges.
Voici l'exemple d’une vulnérabilité IDOR en faisant une référence directe à un objet d’une base de données.
Imaginons un site qui utilise l'adresse suivante pour accéder à la page du compte du client en récupérant des données de la base de données :
https://bad-website.com/customer_account?customer_number=132355
Dans la requête, on remarque ce paramètre “customer_number”, suivi d’un nombre. Il est possible de modifier ce nombre pour avoir accès à la page d’un autre compte et ceci représente une faille IDOR, car on fait directement référence à une ressource à laquelle on ne devrait pas pouvoir accéder. Ici, cette vulnérabilité démontre un exemple d’élévation de privilèges horizontal.
En remplaçant donc le customer_number de 132355 par 123456, on obtient la requête http suivante :
https://bad-website.com/customer_account?customer_number=123456
Et en partant du principe que c’est un numéro de compte valide, on obtient alors accès au compte et donc à certaines informations qui peuvent être sensibles.
CSRF - Client-Side Request Forgery :
Une attaque CSRF est une attaque qui a pour but de manipuler un navigateur web à effectuer une action non voulue sur une application web où un utilisateur est connecté.
L’impact de cette dernière peut engendrer des conséquences dévastatrices pour l’utilisateur. Elle permet par exemple de voler des données sensibles ou des cookies de sessions, d’effectuer des transferts d'argent, de supprimer des comptes, etc…
Imaginons une application web d’achats en ligne, un utilisateur peut généralement créer un compte, le supprimer, ajouter des adresses de livraison ou de facturation,ajouter une carte bancaire etc..
Imaginons maintenant qu’un utilisateur décide de supprimer son compte, il se connectera à son compte et dans les paramètres, il cliquera sur le bouton “Supprimer votre compte”.
Dans le code, appuyer sur ce bouton effectue une requête http au serveur et exécute le code nécessaire à l’action demandée, du type :
POST /supprimer_compte HTTP/1.1
Jusqu’ici c’est un comportement classique et sain d’un serveur http et son client.
Cependant, si maintenant, un attaquant décide de supprimer le compte d’une victime qui ne veut pas supprimer son compte du site, il peut utiliser une attaque CSRF.
A l’aide du schéma ci-dessus, on peut voir les étapes de cette attaque.
Tout d’abord, la victime se connecte à son site d’achats en ligne préférés. L’attaquant va “forger” (construire de toute pièce) un hyperlien qui fait référence à la requête de suppression de compte vu précédemment.
Il va ensuite l’envoyer à sa victime, et lorsque celle-ci cliquera sur l’hyperlien, la requête va s’envoyer au serveur. Etant donné que la victime est connectée à l’application, la requête s’envoie au serveur de l’application web et effectue l’action associée, en l'occurrence la suppression du compte.
Il est également possible pour l’attaquant de leurrer ses victimes sur un site qu’il possède et qui paraît légitime , pour leur faire cliquer sur des hyperliens cachés dans le but d'exécuter ses requêtes, au lieu de directement envoyer l’hyperlien à la victime.
Pour résumer CSRF, Cross Site Request Forgery désigne le fait de forger une requête valide pour un serveur dans le but d’atteindre les ressources associés à un utilisateur qui est connecté à ce dit serveur.
Path Traversal
Path traversal ou directory traversal est une attaque qui consiste à se déplacer dans l'arborescence dans un site pour atteindre des ressources qui ne devraient pas être accessibles normalement.
On pourrait imaginer voir sur un site différentes pages comme port scanners, traceroute, etc…
Ainsi, en imaginant que nous soyons sur un site appelé jeandupont.org, alors par exemple si on vient scanner les ports, on aura une URL du type :
jeandupont.org/port_scanner
Le directory traversal se base sur principe, imaginons un site avec une requête GET dans l’URL, il est possible de rentrer en valeur du paramètre un chemin qui pourrait donner accès à des ressources qui ne devraient pas l’être.
Par exemple, prenons un blog de cuisine qui possède beaucoup de recettes différentes : imaginons qu’il est possible de rechercher une recette, on aura alors une requête qui ressemble à la suivante :
https://cusinemeilleurblog/?recipe=polenta
Un attaquant peut décider de rentrer un chemin qu’il aurait pu identifier à l’aide d’outils comme dirsearch ou dirbuster afin d’accéder à des ressources normalement inaccessibles, il peut alors essayer de rentrer l’URL suivante et ainsi avoir accès au mot de passe du compte admin :
https://cusinemeilleurblog/?recipe=../admin/backup_mdp/.htpasswd

Broken Access Control - Protection contre les failles

Se protéger de ce type d’attaque et renforcer ces vulnérabilités peut être très complexe, car il existe une infinité de scénarios différents, ce qui rend très difficile de tout protéger efficacement. De nos jours, la plupart des frameworks de développement web possèdent des mécanismes de défense.

Génération de tokens sécurisés :
Par exemple, Laravel, un framework PHP, refuse de laisser un formulaire sans la présence d’un token CSRF. Il affiche une erreur et refuse d'exécuter le code sans la présence de ce token.
Un token est une chaîne de caractères et de nombre aléatoires qui ressemble généralement à un hash md5:
1cdaf5dfb46fc4f2bcd0e70225fd69d7
Ce token est ensuite appelé dans toutes les pages du site. Pour générer ce genre de token, il existe des fonctions en php, la fonction ci-dessous est par exemple considérée sécurisée :
md5(bin2hex(openssl_random_pseudo_bytes(6)));
La fonction openssl_random_pseudo_bytes, de son côté, retourne une chaîne pseudo-aléatoire d’octets de taille 6.
La fonction bin2hex convertit ensuite ce chiffre en hexadécimal et md5() hash le contenu en utilisant l’algorithme md5.
Voici le code en PHP :
<?php
if (!isset($_SESSION['jeton'])) {
$_SESSION['jeton'] = bin2hex(openssl_random_pseudo_bytes(6));
}
?>
Ce code signifie que si un jeton n’existe pas pour la session, on en génère un.
Ne pas afficher les IDs :
Il est aussi nécessaire que les développeurs n’affichent pas dans les paramètres des URL des données clés ou privées comme par exemple l’ID d’un compte client.
Il est encore et toujours nécessaire d’effectuer des validations d’entrées des utilisateurs pour filtrer des entrées malicieuses.
Security Misconfiguration

Introduction :

La configuration de la sécurité d’un serveur, d’une application web ou n’importe quel élément de type client-serveur est un élément fondamental. Si on part du principe que chaque utilisateur est potentiellement malfaisant, alors il faut être certain que chaque fonction, chaque attribut, chaque entrée sont bien configurés de manière à ne donner à l’utilisateur que les privilèges auxquels il a droit, en se basant sur la règle du « Least Privileges ».

La mauvaise configuration de la sécurité peut survenir dans n’importe quel niveau d’un système. On peut en trouver dans les services réseau, dans un serveur web, dans un serveur d’application, dans une base de données, un framework, du code personnalisé, ou encore dans des machines virtuelles préinstallées.

C’est pourquoi la mauvaise configuration de la sécurité de n’importe quelle plateforme peut avoir des conséquences désastreuses dans beaucoup de domaines, que ce soit la divulgation d’informations personnelles (adresse, coordonnées bancaires…), la création de comptes malicieux ayant tous les droits, et cela peut aller jusqu’à la destruction des données au sein d’une base de données, menant potentiellement à la perte de milliers voir de millions d’informations pour les bases noSQL des très grands groupes.

Ainsi, la règle de Security Misconfiguration se base plus sur les bonnes pratiques de configuration, de restriction d’accès, de sécurisation d’inputs, d’utilisation de comptes administrateurs… En clair, cette partie s’axe sur des conseils et des façons de faire permettant de mettre en place un serveur / une base de données / une application Web sécurisé(e) en évitant au maximum de laisser la place à des failles.

Pour finir, il faut savoir qu’il existe des scanners automatiques permettant de détecter ces mauvaises configurations, en essayant une grande quantité d’actions sur l’application en question, par exemple l’essai de connexion sur les utilisateurs par défaut, des services inutiles qui sont quand même installés… Cependant, il convient de bien configurer la sécurité par soi-même, et la partie suivante va vous aider dans cette tâche.

Security Misconfiguration - Attaques / failles

Default credentials :
Cette faille de sécurité est généralement la première à être exploitée par une personne malveillante. Cela concerne par exemple, lorsque de la mise en place d’une base de données via PHPMyAdmin, de laisser l’utilisateur root par défaut, ou alors de créer un compte avec user : admin et mot de passe : admin.
Il faut à tout prix soit désactiver ou supprimer les comptes par défaut, ou au minimum modifier les valeurs attribuées au nom d’utilisateur et au mot de passe. Cross-Site Scripting XSS

Introduction :

XSS ou Cross-Site Scripting est une vulnérabilité web qui a pour but d’injecter des scripts javascript par le biais d’un formulaire ou dans les requêtes http sur une page web ou sur n’importe quel élément html dans lequel on peut y entrer des inputs, dans le but de compromettre toutes les actions que les utilisateurs peuvent avoir sur le site victime. Cette attaque peut causer différents problèmes, comme l'exécution de code arbitraire ou le vol d’informations critiques (données de carte bancaire, adresses mail, mots de passe, identifiants de connexion, etc…).

Le schéma ci-dessous explique plus clairement les étapes qui peuvent composer une attaque XSS, ces étapes peuvent êtres sensiblement modifiées selon la vulnérabilité exploitée. Il s’agit ici d’une vulnérabilité Reflected XSS, les différentes vulnérabilités XSS seront expliquées par la suite.

On peut retrouver 3 catégories dans cette attaque qui ont chacune leurs spécificités :
● Reflected XSS :
Reflected XSS est la vulnérabilité la plus simple parmi les 3 citées. Elle a lieu lorsqu’un attaquant manipule une requête HTTP et y insère du code malicieux. Par exemple, supposons un site web qui propose de pouvoir effectuer une recherche :
https://insecure-website.com/search?term=gift
Imaginons que nous rentrons le terme “gift” pour notre recherche, le site affichera alors le texte suivant:
<p> You searched for : gift </p>
Pour l’instant rien d'extraordinaire, mais si le site n’effectue pas d’opérations de validation sur cet input, alors un hacker peut en profiter pour rentrer du code javascript malicieux.
La requête précédente pourrait donc devenir la suivante:
https://insecure-website.com/search?search=<script> /* Vol de cookies */ </script>
De ce fait, si un autre utilisateur utilise cette requête html alors le hacker récupérera les cookies de sessions de cet utilisateur. Il suffit que le hacker poste un message sur les réseaux sociaux : “Regardez cette superbe réplique de l’anneau du seigneur des anneaux, elle n’est pas super chère et c’est de l’or blanc gravé !! : https://insecure-website.com/search?search= <script> /* Vol de cookies */ </script> “
Les gens intéressés par l’article qui cliqueront sans faire attention se feront avoir. Vous l’aurez compris, l’impact d’une telle vulnérabilité est plutôt important : si l’attaquant arrive à faire s'exécuter le script dans le navigateur de la victime, il peut complètement compromettre la victime.
Comment trouver cette vulnérabilité ? Plusieurs solutions existent :
- Tester toutes les zones d’input, il faut tester séparément toutes les zones d’entrées où l’on peut effectuer une requête http sur le site web.
- Pour chaque point d'entrée, entrer des valeurs alphanumériques aléatoires, une suite de 8 caractères alphanumériques est une bonne pratique pour tester si cette chaîne de caractères est reflétée sur le site ou non.
- Après avoir entré les valeurs, vérifier où est reflétée la chaîne de caractères, il faut savoir si c’est dans des balises de textes, dans des listes, dans des balises de scripts, etc…, pour comprendre dans quel contexte le script pourrait s'exécuter.
- Si votre premier test d’attaque XSS est bloqué ou modifié par le navigateur, essayez différentes charges utiles, il n’y a pas qu’un seul moyen d'exécuter des scripts.
● DOM XSS :
DOM XSS, pour Domain Object Model based XSS, est une attaque XSS qui vise directement le DOM d’un site web en y injectant du code malicieux ou en le modifiant d’une quelconque manière qui profiterait à l’attaquant. Le DOM est la représentation des objets qui constituent l’architecture et le contenu d’une page web. Le DOM permet de lier la programmation d’une page web avec l’apparence et l’architecture de cette page web.
Un attaquant peut utiliser différents objets DOM pour créer une attaque XSS. Généralement, les objets document.url, document.location et document.referrer sont utilisés pour réaliser ces attaques.
L’exemple suivant explique une attaque DOM XSS basique.
Imaginons une page web personnalisée en fonction du nom d’utilisateur, le code de la page web est le suivant :
<html>
<head>
<title>Page personnalisée</title>
.......
</head>
<script>
var pos = document.URL.indexOf(“context=”)+8;
document.write(document.URL.substring(pos,document.URL.length));
</script>
.......
</html>
Si l’utilisateur Marie a une page personnalisée sur ce site, l’url de sa page sera :
http://www.example.com/userdashboard.html?context=Marie
Maintenant imaginons une attaque DOM-based XSS, elle pourra être exécutée de la manière suivante :
http://www.example.com/userdashboard.html?context=<script> Malicious Code </script>
Le navigateur de la victime reçoit l’URL, commence à construire le DOM et parse l’URL. Le navigateur parse la page Html, atteint le premier script, et extrait le code malicieux contenu dans “context”. Une fois extrait, le navigateur l’ajoute à la page dans le DOM, le comprend comme du code javascript et l’exécute.
A la différence de l’attaque précédente, l’attaque modifie le DOM de la page pour exécuter le code malicieux et n’est cette fois-ci pas directement inscrit dans le code source de la page. De plus, ce code n’atteint jamais le serveur et reste toujours côté client !
Ici, la vulnérabilité est différente par rapport à la précédente, car l’attaque exploite une manipulation non sécurisée du DOM et des propriétés du code côté client. A savoir que l’attaquant peut, comme pour l’attaque précédente, encoder le payload de la requête pour ne pas que ça paraisse trop évident.
● Stored XSS :
Pour finir, nous parlerons de la dernière catégorie de vulnérabilité XSS, Stored XSS aussi connue sous le nom de XSS persistant.
La spécificité de cette vulnérabilité est plus simple à comprendre et différencier par rapport aux autres. Imaginons un nouveau site non sécurisé qui permet d’écrire des commentaires sur des forums et qui sont sauvegardés dans une base de données.
Le mode opératoire reste toujours plus ou moins le même, vous l’aurez compris. Sur ce forum, un utilisateur rentre le commentaire suivant : “Super cool ce forum”. Ce commentaire restera dorénavant affiché sur le forum et sera sauvegardé dans une base de données.
Toujours en partant du principe que le serveur ne fait aucune opération de validation de données, un attaquant pourrait remplacer le simple commentaire en un script javascript malicieux. Ainsi, le commentaire pourra être de la forme suivante :
<script> Malicious Code </script>
L’avantage de cette technique par rapport aux deux autres est que cette fois-ci, l’attaquant n’a pas besoin d’induire en erreur d’autres utilisateurs en les faisant se connecter à un site web vulnérable avec un url contenant le code javascript. Ici, le code malicieux reste ancré sur le forum et donc chaque utilisateur qui se connecte à ce forum l’exécutera et activera le code malicieux.

XSS - Protection contre les failles :

Afin de se protéger contre les attaques XSS, il est nécessaire d’effectuer des validations de données et encoder les données reçues dans les inputs pour faire en sorte que le navigateur ne reconnaisse pas les données malicieuses comme du code. Il existe plusieurs méthodes pour se défendre contre ces attaques, il est préférable d’en utiliser le plus possible, une seule méthode ne suffit pas pour se protéger des attaques XSS. Il faut savoir qu’elles sont vouées à être complétées, ce ne sont que les bases.

Comme nous allons le voir par la suite, les attaquants trouvent toujours un moyen d’esquiver les filtres ou les types d’encodage. Il est donc intéressant de comprendre comment ils les esquivent pour continuer d’améliorer les mécanismes de défense.

Filtrer :
Il est nécessaire de filtrer au maximum les données reçues au niveau des inputs. On peut par exemple imaginer un formulaire d’inscription qui demande identifiant, mot de passe et adresse mail.
Pour l’input de l’identifiant, on peut faire en sorte de ne pas accepter les caractères spéciaux tels que ‘<’ ou '>' : pour cela, il est possible d’utiliser la fonction php filter_input().
Documentation de la fonction: https://www.php.net/manual/fr/function.filter-input.php
Pour l’input mot de passe, on peut aussi faire en sorte de ne pas accepter les caractères spéciaux, mais cela peut poser problème pour des personnes qui utilisent des mots de passe à sécurité forte, ainsi il serait préférable d’utiliser une autre méthode que l’on verra par la suite.
Pour l’input de l’adresse mail, on peut filtrer à l’aide d’expressions régulières qui empêcheraient à l’attaquant d’écrire des scripts puisque l’expression régulière demande une chaîne de caractères qui ressemble à une adresse mail, c'est-à-dire une suite de lettres qui précède un @, suivi du nom de domaine du mail, “gmail.com”, “orange.fr”, etc…

Utiliser les en-têtes de réponse :
L’utilisation des headers Content-Type ou X-Content-Type Options permet de mitiger les attaques XSS en spécifiant de quels types doit être la requête.
L'en-tête Content-Type sert à indiquer le type MIME(Ex: text/plain, application/json, image/jpg, etc…) de la ressource.
Dans les réponses, une en-tête Content-Type indique au client le type de contenu réellement renvoyé. Il peut arriver que les navigateurs cherchent à détecter le type MIME du contenu en l’inspectant au cas où un attaquant malveillant voudrait envoyer un fichier d’un autre type que celui spécifié. Pour empêcher ce comportement, on peut paramétrer l'en-tête X-Content-Type-Options avec la valeur nosniff. La valeur nosniff permet de bloquer une requête si la destination de la requête est de type “script” ou “css”.
Content Security Policy :
Une Content Security Policy (CSP) ou stratégie de sécurité du contenu permet d'améliorer la sécurité des sites web en permettant de détecter et réduire certains types d'attaques, dont les attaques XSS (Cross Site Scripting) et les injections de contenu. Ces attaques peuvent être utilisées dans divers buts, comme le vol de données, le défacement de site ou la diffusion de malware.
CSP permet aux administrateurs système de réduire ou éliminer les moyens de réaliser des attaques XSS en permettant de spécifier les domaines autorisés à fournir des scripts pour la page visitée. Un navigateur compatible avec CSP n'exécute que les scripts provenant d'une origine autorisée par les règles CSP reçues et ignore ceux qui ne sont pas autorisés. On peut ainsi bloquer les domaines non autorisés, les scripts inline (inclus dans une page HTML) ou associés à des événements via les attributs HTML dédiés.
Pour un niveau de protection le plus élevé possible, un site qui voudrait qu'aucun script ne puisse être exécuté peut simplement désactiver l'exécution de tout script.
Pour activer CSP, il faut configurer le serveur web afin d'ajouter une en-tête (header) HTTP Content-Security-Policy aux réponses. Vous pouvez rencontrer des documents qui mentionnent X-Content-Security-Policy comme en-tête, il s'agit d'une version obsolète qu'il n'est plus utile de supporter.
Une autre possibilité consiste à utiliser l'élément HTML <meta> pour configurer la règle, par exemple :
<meta http-equiv="Content-Security-Policy" content="default-src 'self'; img-src https://*; child-src 'none';">

Introduction :

On imagine souvent que les hackeurs travaillent d’arrache-pied afin de trouver de nouvelles failles te de créer des scripts pour aller les exploiter spécifiquement, et que le moindre nouveau bout de code aura une faille qu’il saura trouver.

Prenons un exemple pour mieux comprendre cette vulnérabilité : imaginons une application Web basée sur un serveur Apache s’appuyant sur une SBGD Oracle. Pour chaque élément mentionné, il existe malheureusement (et heureusement aussi) des vulnérabilités connues. Malheureusement, car un hackeur peut simplement faire des recherches et trouver facilement un moyen de trouver une faille, mais aussi heureusement, car il existe aussi beaucoup de documentation pour se prémunir d’une faille de ce genre.

Ainsi, si un hackeur arrive à trouver le type de serveur contenant les données ainsi que la version, le système devient vulnérable, à moins d’avoir été sécurisé en amont lors de la mise en place de l’application Web.

Using Components With Known Vulnerabilities – Attaque / failles

Frameworks :
Les framework, bibliothèques et librairies utilisées dans une application sont, pour la plupart, exécutées avec tous les privilèges. Un hackeur peut simplement utiliser des outils de scan automatisés (détaillés dans la partie sur la protection contre les failles) ou simplement analyser l’application au travers du code source afin de trouver le type de framework utilisé.
Si ce framework est connu pour avoir des failles, le hackeur les trouvera lors de recherches sur Internet, et il pourra alors l’exploiter à son avantage.
Pour trouver ces frameworks, les hackeurs cherchent principalement des éléments HTML connus, essaient de forcer l’application à renvoyer un certain type d’erreur, ou encore forcent certaines recherches qui ne sont potentiellement pas prévues initialement pour essayer de trouver le nom des dépendances, voir à quelles fonctions font référence les messages d’erreur…
Components With Full Privileges :
Les composants d’une application (fonction, frameworks, APIs, librairies..) sont généralement exécutés via un niveau de privilèges égal à celui de l’application elle-même. Ainsi, s’il existe des faiblesses au sein d’un seul de ces composants, un hackeur pourrait l’utiliser à son avantage.
Par exemple, pour des classes Java, si celles-ci possèdent toutes des héritages entre différentes classes sans abstraction quelconque, trouver une brèche dans n’importe quelle classe peut permettre à un hackeur d’utiliser toutes les autres classes qui en découlent.
IoT :
L’internet des objets (IoT) contient l’ensemble des appareils connectés possédant un système d’exploitation, aussi rudimentaire soit-il. On compte ainsi énormément d’objets différents, allant du frigo et de la machine à pain au téléphone ou la télé intelligente. Ces objets ont une réelle utilité au quotidien en facilitant leur utilisation.
Cependant, leur utilité facile et la basse complexité de leurs OS les rend très fragiles : généralement pas sécurisés (car compliqués à sécuriser), il peut être relativement facile de prendre contrôle de tels objets à distance. Le problème étant qu’il existe des outils de scan permettant aux hackeurs de trouver les types d’appareils qui souffrent de certains types d’attaques.
Par exemple, d’après le site OWASP sur cette attaque, l’outil Shodan IoT search engine permet à n’importe quel utilisateur de connaître quel type d’appareil est encore vulnérable à l’attaque Heartbleed, la localisation dans le monde, etc… Alors que cette faille a été patchée en 2014.
Pour explication, l’attaque Heartbleed est une faille basée sur la librairie OpenSSL cryptography, et plus précisément sur l’implémentation du protocole TLS, permettant de transporter les paquets de données de façon sécurisée.
Cette faille s’appuie sur l’extension Heartbeat (d’où son nom), qui permet de sécuriser le lien entre un client et un serveur en envoyant un Heartbeat Request message, composé d’une payload accompagnée d’un nombre entier sur 16bits. Le problème est que les versions d’Open SSL ayant une vulnérabilité possédaient un buffer de mémoire, qui pouvait donc renvoyer un message correspondant à la taille requise par le message entré.
Ainsi, il était possible de faire afficher au serveur ce qui se trouvait dans son buffer en demandant en entrée de retourner une chaîne de caractère très grande.
Per exemple, si l’utilisateur Paul demande au serveur de lui renvoyer le message de 7 lettres « bonjour » pour savoir s’il fonctionne, dans une utilisation normale, le serveur lui renverra le message de 7 lettres « bonjour ».
Cependant, dans le cas de l’exploitation de la faille, si notre utilisateur Paul demande au serveur de lui renvoyer le message cette fois de 500 lettres « bonjour », le serveur renverre une chaîne de caractères contenant 500 lettres à partir du mot « bonjour ».
On pourrait donc se retrouver avec un message du type « bonjour. L’utilisateur Jacques vient de se connecter via le nom d’utilisateur : jacques1u et le mot de passe : jacques. Mise à jour de l’OS vers la version 1.25.352… » : l’attaquant peut donc avoir accès à des informations extrêmement sensibles grâce au buffer, et donc à la faille. Il en existe évidemment des milliers d’autres.

Using Components With Known Vulnerabilities – Attaque / failles

Inventaire / Téléchargement :
L'administrateur du site doit faire un inventaire continu du système client-serveur : en faisant un inventaire régulier, il est possible de voir qui s'est connecté, quelles actions ont été entreprises (le tout via les logs). Cela permet de retracer les actions entreprises sur le serveur et de détecter d'éventuelles intrusions.
Télécharger des composants provenant de sources et sites sûr(e)s, afin de ne pas être surpris par du code malicieux ou des exécutables corrompus lors de l'installation des différents éléments. Il faut par exemple essayer de télécharger les composants sur des sites disposant de certificats de sécurité (https, ssl...).
Utilisation de Frameworks :
Eviter d’utiliser des frameworks, car ils donnent généralement de base des privilèges extrêmement élevés, ou alors reprendre toutes les classes, les fonctions afin de redéfinir la limite de leur utilisation (abstraction de classes dans Java par exemple, afin de limiter l’action d’un client sur la partie métier d’un service / d’une application).
Avoir un plan de structuration :
Cela signifie que, suivant les services utilisés pour le serveur et la/les base(s) de données, il faut dans un premier temps se renseigner en profondeur sur les différentes brèches qui existent et trouver un moyen de s’en prémunir.
Il faut également pouvoir scanner son application le plus souvent possible, pour pouvoir patcher les éléments pour lesquels une nouvelle faille a été trouvée (soit en interne soit en reprenant des patchs de corrections disponibles sur internet), ou alors reconfigurer certains éléments. Il faut donc effectuer une surveillance régulière.
Outils de scan :
Il existe pléthore d’outils de scan, payants ou non, accessibles sur Internet. Parmi eux, deux sont à noter car ils puisent leurs informations dans les deux plus importantes bases de données de vulnérabilités disponibles (qui seront décrites juste après) :
Gemnasium
Gemnasium est un outil commercial qui utilise sa propre base de données qui prend appui sur plusieurs sources. Cet outil a recours à un algorithme permettant de tester les meilleures combinaisons de dépendances au-lieu de toutes les tester, ce qui permet de faire les scans plus vite et plus efficacement. Cet outil supporte également plusieurs langages (Ruby, NPM (Javascript), PHP, Python et Bower (Javascript)).
Dependency-check
Cet outil en ligne de commande open-source a été créé par OWASP. Bien maintenu, et s’appuyant sur la base de données NIST NVD, il supporte Java, .NET, Javascript et Ruby. Etant open-source, il est donc gratuit.
Bases de données de vulnérabilités :
Il existe des bases de données reprenant énormément de vulnérabilités existantes, parmi lesquelles la CVE (Common Vulnerabilities and Exploits) et la NVD (National Vulnerability Database).
Elles contiennent donc des failles concernant la plupart des logiciels, des serveurs, des services existants. Cela représente une mine d’informations colossale qui permet donc à des personnes souhaitant se défendre contre de potentielles attaques d’avoir des explications, des aides à la solution et des outils pour s’aider.
Les deux bases marchent en étroite collaboration : la base CVE permet de trouver une faille via des mots clés ou des ID si connus, afin de filtrer vers la bonne faille, tandis que la NVD, une fois la faille identifiée, donne accès à des outils, des conventions et des solutions à ces problèmes.
